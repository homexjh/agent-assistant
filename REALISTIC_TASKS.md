# aiagent 实际任务拆解 (基于当前架构)

## 当前架构现状

### 已实现的核心 (现有代码)
```
aiagent/
├── agent.py           # 简单Tool-Use循环 (206行)
├── subagent.py        # threading子Agent (117行)
├── subagent_registry.py   # JSON持久化 (139行)
├── subagent_tools.py  # 4个子Agent工具 (260行)
├── skills.py          # Frontmatter扫描 (101行)
├── workspace.py       # MD文件加载 (59行)
├── serve.py           # HTTP+SSE (509行)
├── main.py            # CLI入口 (54行)
└── tools/
    ├── __init__.py    # 简单注册表 (94行)
    ├── types.py       # TypedDict定义 (40行)
    ├── exec.py        # Shell执行 (67行)
    ├── file.py        # 读写编辑 (429行)
    ├── web.py         # 抓取搜索 (186行)
    ├── browser.py     # Playwright (413行)
    ├── cron.py        # 定时任务 (401行)
    ├── process.py     # 后台进程
    ├── memory.py      # 记忆搜索
    ├── image.py       # 图片分析
    ├── pdf.py         # PDF提取
    ├── tts.py         # 语音合成
    └── git_enhanced.py    # Git增强
```

### 当前技术特点
- 工具注册: 简单字典 `_registry: dict[str, RegisteredTool]`
- 工具定义: TypedDict手写Schema
- 子Agent: threading + queue.Queue
- Skill: 简单YAML frontmatter解析
- 持久化: JSON文件
- 通信: 内存queue + 磁盘JSON

### Week 2: 子Agent系统加固 (Day 6-10)

#### Day 6: 子Agent生命周期细化
**现状**: CREATED → RUNNING → COMPLETED，无中间状态
**任务**:
- [ ] **状态扩展**: 更细粒度状态

- [ ] **状态钩子**: 状态变更回调

- [ ] **状态查询API**: 更灵活查询

## 建议实施策略

### 方案A: 渐进增强 (推荐)
1. 先完成第一阶段(20天)，得到稳定可用版本
2. 根据用户反馈选择升级方向
3. 逐步引入高级特性

### 方案B: 直接高级
1. 直接按完整架构实现
2. 开发周期长(70天)
3. 风险较高

### 方案C: MVP优先
1. 只做第一阶段核心(10天)
2. 快速验证需求
3. 再决定后续投入

**推荐方案A**，先让当前架构达到生产可用，再逐步增强。
